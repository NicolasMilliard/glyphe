# Glyphe CLI Plan

The CLI should stay faithful to the core product principle: generate code the user owns. It should not hide animation behavior behind a runtime package unless the user explicitly chooses that path later.

## Package Plan

Initial public CLI package name: `glyphe`.

Primary executable name:

- `glyphe`

Because the unscoped `glyphe` package name is available on npm, the best user-facing flow is:

```sh
bunx glyphe init
bunx glyphe add spinner/braille
bunx glyphe add loader/waveform --react
bunx glyphe add text/typewriter --tailwind
bunx glyphe registry list
bunx glyphe generate --frames "⠋ ⠙ ⠹ ⠸ ⠼"
```

The package should expose a `bin` entry named `glyphe`:

```json
{
  "name": "glyphe",
  "bin": {
    "glyphe": "./dist/index.js"
  }
}
```

The future `@glyphe/*` packages should be treated as library/package boundaries, not as the primary executable package:

- `@glyphe/core`: shared types, config, and low-level helpers
- `@glyphe/registry`: bundled official registry data
- `@glyphe/generator`: CSS, React, and Tailwind generation logic
- `@glyphe/react`: optional React helpers or adapters, if needed
- `@glyphe/tailwind`: optional Tailwind plugin, if needed
- `@glyphe/cli`: optional internal workspace package name only if the repo later wants the CLI source package scoped while publishing the unscoped `glyphe` wrapper

Recommended publishing model: publish the executable as `glyphe`; publish reusable libraries as `@glyphe/*` only when they become useful outside the CLI and website.

Initial implementation should live in a future package boundary:

```txt
packages/
  glyphe/
    src/
      index.ts
      commands/
        init.ts
        add.ts
        registry.ts
      lib/
        config.ts
        detect-project.ts
        generate-files.ts
        write-file.ts
  core/
  registry/
  generator/
```

The CLI should import shared registry/generator logic after those modules are extracted from the app. Until then, a local prototype can import from `src/registry` and `src/generator` to validate the workflow.

## Config Schema

Default config filename: `glyphe.config.json`.

Conceptual schema:

```ts
type GlypheConfig = {
  $schema?: string
  style: "default"
  tsx: boolean
  tailwind: boolean
  registry: {
    source: "local" | "remote"
    url?: string
  }
  output: {
    components: string
    css: string
    tailwind?: string
  }
  aliases?: {
    cn?: string
  }
}
```

Default config:

```json
{
  "$schema": "https://glyphe.dev/schema.json",
  "style": "default",
  "tsx": true,
  "tailwind": true,
  "registry": {
    "source": "local"
  },
  "output": {
    "components": "src/components/glyphe",
    "css": "src/styles/glyphe.css",
    "tailwind": "src/styles/glyphe-tailwind.css"
  },
  "aliases": {
    "cn": "@/lib/cn"
  }
}
```

## `init` Command

Command:

```sh
glyphe init
```

Responsibilities:

- detect package manager
- detect React and TypeScript
- detect Tailwind usage
- create `glyphe.config.json`
- create output folders when requested
- print next-step commands

Flags:

- `--yes`: accept defaults
- `--cwd <path>`: run against another project directory
- `--no-tailwind`: skip Tailwind output defaults
- `--dry-run`: print files that would be written

`init` should not install runtime dependencies in the MVP.

## `add` Command

Command:

```sh
glyphe add spinner/braille
```

Responsibilities:

- read config
- resolve registry item
- generate CSS
- optionally generate React component
- optionally generate Tailwind output
- write files into configured output paths
- preserve existing user edits unless overwrite is explicit

Flags:

- `--react`: generate a React component
- `--css`: generate CSS output
- `--tailwind`: generate Tailwind output
- `--all`: generate every supported output
- `--name <ComponentName>`: override React component name
- `--dry-run`: show planned writes
- `--force`: overwrite managed files
- `--cwd <path>`: run against another project directory

Default MVP behavior should generate CSS and React when the project is React/TSX, and plain CSS otherwise.

## Dry Run Behavior

Dry run should never write files.

It should print:

- resolved config path
- resolved registry item
- output files that would be created
- output files that would be changed
- conflicts that require `--force`
- generated command suggestions

Dry run should be available for both `init` and `add`.

## File Overwrite Behavior

Generated files should include a small managed marker:

```txt
/* Generated by Glyphe. You own this file. */
```

Overwrite rules:

- create missing files
- append to CSS files only when a matching generated block is absent
- replace generated blocks only when the marker and animation slug match
- refuse to overwrite unmarked files unless `--force` is passed
- show a clear conflict message with the file path and suggested flag

The CLI should avoid destructive behavior by default.

## Registry Fetch Behavior

MVP registry source: local bundled registry.

Future registry sources:

- remote official registry
- user-provided registry URL
- local registry path

Resolution order:

1. explicit `--registry <path-or-url>`
2. `glyphe.config.json` registry source
3. bundled registry

Remote fetches should cache registry data and validate entries with the shared Zod schema before generation.

## Local Registry Install Prototype

Status: implemented as a local spike in `cli/`.

Prototype scope:

```sh
glyphe add spinner/braille --all --dry-run
glyphe add spinner/braille --all
```

Exit criteria:

- resolves one local registry item
- generates CSS, React, and Tailwind output
- writes to a temporary fixture project
- refuses to overwrite unmarked files
- supports dry-run output
- generated output matches website generator output

This should be implemented only after package boundaries are planned or after a small CLI spike is explicitly approved.

Current prototype command:

```sh
bun ./cli/index.ts add spinner/braille --all --dry-run
bun ./cli/index.ts add spinner/braille --all --cwd /path/to/fixture
```

The prototype intentionally writes through a planned-file layer before touching disk. It supports managed markers, dry runs, forced React overwrites, CSS/Tailwind block replacement, and refusal to overwrite unmanaged component files.
